# xlsx2md 仕様

## 1. 文書概要

`xlsx2md` は、Excel 形式の設計書ファイル (`.xlsx`) を解析し、シート単位の Markdown 文書へ変換するツールである。

本ツールの目的は Excel の見た目を再現することではなく、設計書としての構造・意味・追跡可能性を保ちながら、生成 AI が入力として処理しやすい Markdown に変換することである。

本ツールは、単なる `Excel -> Markdown` ではなく、`設計書構造抽出 -> Markdown` として設計する。

本書は上位仕様および設計方針を扱う。現行実装の詳細な挙動は `xlsx2md-impl-spec.md` を参照すること。

文書の役割分担は次の通りである。

- `README.md`
  - 入口説明と全体像
- `xlsx2md-spec.md`
  - 人間が意図、対象範囲、設計方針を理解するための橋渡し文書
- `xlsx2md-impl-spec.md`
  - 現行実装の詳細挙動、内部状態、再実装・保守のための文書
- `xlsx-formula-subset.md`
  - 数式対応範囲の補助文書

## 2. 適用範囲

### 2.1 対応入力形式

- `.xlsx`

### 2.2 非対応入力形式

初版では以下を対象外とする。

- `.xls`
- `.csv`

### 2.3 想定対象文書

本ツールは、以下のような設計書系 Excel を主対象とする。

- プログラム設計書
- 画面設計書
- バッチ設計書
- IF 設計書
- 項目定義書
- テーブル定義書
- テストケース定義書

## 3. 基本設計方針

### 3.1 出力目的

出力 Markdown は、以下を満たすことを目的とする。

- 生成 AI が入力として解釈しやすい
- 人間が読んで意味をある程度把握できる
- トレーサビリティを確保できる
- 元の Workbook / Sheet / Range / Anchor を追跡できる

### 3.2 出力方針

- 出力はシート単位とする
- 各シートは 1 つの Markdown ファイルとする
- Workbook 名は必須メタ情報として保持する
- 表検出は、値または罫線を持つセル群の連結成分を起点とし、罫線・密度・ヘッダ性などを手掛かりに表候補を推定する
- 見た目の忠実再現ではなく、構造と意味の保存を優先する
- グラフや図形も、画像化やピクセル再現ではなく、意味情報や raw metadata のテキスト化を優先する
- Markdown 本文の既定値は Excel 表示寄りとする
- Excel 表示寄りとは、人間が既存の設計書 Excel 上で見ている値を生成 AI と共有しやすい形を優先することを意味する
- ただしピクセル単位の見た目再現は目的とせず、表示値・構造・トレーサビリティの両立を優先する
- 装飾、図形、広い余白、複雑な merge、カレンダー配置のようなレイアウト中心の文書については、Workbook 上の見た目をそのまま再現するのではなく、セクション・表・リスト・画像へ分解して表現することを優先する
- 特に、カレンダー、プランナー、ダッシュボード、ポスター風テンプレートのようなシートは、通常の表検出だけで自然に表現できない場合がある

### 3.3 実装方針

- 配布形態は Single-file Web App とする
- 画面 UI は `lht-cmn/` を正本とし、原則 `lht-*` コンポーネントを利用する
- 実装の正本ソースは TypeScript とする
- 自動テストを前提とし、主要な変換ロジックには継続的にテストを追加する

## 4. 出力単位およびファイル構成

### 4.1 出力単位

- 1 Workbook を入力とする
- 1 Sheet ごとに 1 Markdown ファイルを出力する

### 4.2 出力ファイル例

入力:

- `sales.xlsx`

出力:

- `sales_001_Summary.md`
- `sales_002_Detail.md`

`raw` / `both` モード時の出力例:

- `sales_001_Summary_raw.md`
- `sales_001_Summary_both.md`

### 4.3 出力ディレクトリ構成

出力ディレクトリは以下の構成を基本とする。

```text
output/
  sales.md
  assets/
    Summary/
      image_001.png
      image_002.png
    Detail/
      image_001.png
```

- ブラウザ UI の `Markdown+assets ZIP保存` では、既定で Workbook 単位の連結 Markdown 1 本と `assets/` をまとめて出力する
- シートごとの分割 Markdown を ZIP に含める方式は、現時点では標準挙動としない

### 4.4 ファイル命名

- Markdown ファイル名には Workbook 名、シート順、シート名を利用する
- シート順は Workbook 内の物理順序に従う
- シート名にファイル名として不適切な文字が含まれる場合はサニタイズする
- サニタイズ時は、ファイルシステム上で不安定な記号、空白、連続区切りを整理して保存名へ反映してよい
- 例: シート名 `A B-東京&大阪.01` は、保存名で `A_B-東京_大阪.01` のように変換してよい
- 出力モードが `display` 以外の場合、Markdown 保存名には `_raw` または `_both` のサフィックスを付与してよい
- 全シート連結保存や ZIP 保存でも、`display` 以外のモードでは同様にモード名サフィックスを付与してよい
- 連結 Markdown 保存名は `workbook.md` を基本とし、モード付きでは `workbook_raw.md` や `workbook_both.md` としてよい

## 5. Markdown 出力構造

### 5.1 基本構造

各 Markdown は、以下の構造を基本とする。

```markdown
# Summary

## ソース情報
- Workbook: sales.xlsx
- Sheet: Summary

## 本文

このシステムは XXX です。
YYY システムとの連携を ZZZ です。

### 表001 (C3-F5)

| Region | Sales | Target | Status |
| --- | --- | --- | --- |
| East | 120 | 100 | Good |
| West | 90 | 100 | Below |

### 画像001 (C7)
- File: assets/Summary/image_001.png

![image_001](assets/Summary/image_001.png)
```

### 5.2 出力要素

各 Markdown には以下を含める。

- シート名
- ソース情報
  - Workbook 名
  - Sheet 名
- 地の文
- 必要に応じて表
- 必要に応じて画像情報
- 必要に応じてグラフ情報
- 必要に応じて図形情報

### 5.3 ソース情報

トレーサビリティ確保のため、各 Markdown 冒頭に以下を必ず出力する。

- Workbook 名
- Sheet 名

Workbook 名は省略しない。

### 5.4 内部 ID の扱い

内部的な表 ID や画像 ID を保持してよいが、本文の主見出しでは必須としない。必要な場合のみ補助メタ情報として扱う。

### 5.5 グラフブロック

- `drawing` 内の chart 参照は、画像ではなく `グラフブロック` として扱ってよい
- 初期実装では、少なくとも次を Markdown に出力してよい
  - anchor
  - タイトル
  - 種別
  - 系列名
  - category range ref
  - value range ref
- 複合グラフの場合、種別は `棒グラフ + 折れ線グラフ (複合)` のように複数種類を併記してよい
- 副軸系列を判定できる場合は、系列ごとに `副軸` であることを出力してよい
- グラフは当面、SVG 化せず、意味情報のテキスト化を標準とする
- 将来の再検討余地は残すが、現時点ではタイトル・系列・参照範囲・副軸の Markdown 化を優先する

### 5.6 図形ブロック

- `drawing` 内の DrawingML 図形 (`xdr:sp`, `xdr:cxnSp` など) は、初期実装では `図形ブロック` として扱ってよい
- 図形ブロックでは、要約済みの意味ラベルよりも、タグ名・属性名を保った raw metadata のテキスト化を優先してよい
- Markdown では、親階層を 4 スペース字下げで表現することで、`xdr:*` や `a:*` の階層を可読に保ってよい
- raw metadata の出力は維持したまま、単純図形 (`rect`, `line`, `arrow`, `text box` など) については SVG アセット参照を併記してよい
- 少なくとも次の情報を出力してよい
  - anchor (`from` / `to`, `col`, `row`, `colOff`, `rowOff`)
  - 図形要素名
  - `cNvPr@id`, `cNvPr@name`
  - `prstGeom@prst`
  - `xfrm/off`, `xfrm/ext`
- テキスト (`a:t`)
- 線や塗りに関する属性
- 接続先 ID が取得できない場合は、無理に補完せず出力しなくてよい
- SmartArt は現時点では意味解釈対象にせず、fallback として扱ってよい
- SmartArt を除く単純図形 (`rect`, `line`, `arrow`, `text box` など) は、SVG 化して添付してよい
- 複雑図形や SmartArt については、当面 raw metadata と fallback を優先し、完全な SVG 再現は対象外としてよい

## 6. 地の文抽出仕様

### 6.1 基本方針

- 表候補に含まれなかった非空セルを地の文候補として扱う
- 地の文候補は、近接したセル群を段落ブロックとしてまとめる
- 1 セルごとに独立した文章としては扱わず、行方向・列方向の近接性を考慮して連結する

### 6.2 地の文ブロック化ルール

- 同一行で隣接する文字セルは半角スペース区切りで連結する
- 連続する行が近い開始位置から始まる場合は同一段落候補として扱う
- 1 行以上の明確な空白行がある場合は段落を分割する
- 罫線で囲まれた領域は地の文ではなく表候補を優先する
- 表候補外の非空セル群は、上から下、左から右の順で Markdown へ出力する

### 6.3 初版の扱い

- 初版では「見出し」「注記」「本文」を細かく分類しすぎず、まずは地の文ブロックとして出力する
- 将来必要であれば、見出し候補や注記候補の分類ルールを追加する

### 6.4 リストブロックの扱い

- 地の文候補のうち、縦方向に独立文が連続する領域は、段落ではなくリストブロックとして扱ってよい
- リストブロック判定は、特定語ではなく構造を主手掛かりとする
- 先頭付近に `完了 / タスク` のような見出し語がある場合は補助情報として使ってよいが、必須条件にはしない

主な判定観点:

- 同じ左端付近から始まる短文が連続している
- 各行が独立した項目文として自然であり、連結すると段落として不自然になる
- 状態列、記号列、チェック列、番号列のような短い補助列を伴うことがある
- 罫線による強い表構造は持たない、または表スコアが低い

Markdown 出力方針:

- リストブロックは Markdown の箇条書きへ変換してよい
- 完了状態やチェック状態を示す短い補助値がある場合は、`- [x] ...` または `- [ ] ...` のように出力してよい
- 補助値がない場合は、通常の `- ...` 箇条書きとして出力してよい

### 6.5 地の文とリストの優先関係

- 行が分かれているだけでは、直ちに別段落やリストとはみなさない
- まず表候補を除外し、その後に段落ブロックとリストブロックを判定する
- 段落として連結した方が自然な場合は地の文ブロックを優先する
- 1 行ずつ読む方が自然な場合はリストブロックを優先する

### 6.6 将来構想: セクション分割ブロック

この節は将来構想を記述する。現行実装の正本は、通常の表検出と地の文抽出を優先する `xlsx2md-impl-spec.md` 側の記述とする。

- レイアウト中心のシートでは、表検出の前後で `セクション分割ブロック` を導入してよい
- セクション分割ブロックとは、見出し、装飾、余白、画像、複数表のまとまりにより、人間が 1 つの意味単位として読む領域を指す
- 目的は Excel の見た目再現ではなく、巨大な 1 表や大量の `[MERGED←]` へ吸われることを避け、Markdown 上で意味のまとまりを保つことである

主な適用対象:

- プランナー
- ダッシュボード
- カレンダー
- ポスター風テンプレート
- 入力パネル + 集計表 + 画像が混在するシート

主な判定観点:

- 太字見出しや装飾見出しの存在
- 見出しの上下に大きな余白や区切り線がある
- 画像群と表群が視覚的に分離している
- merge が非常に広く、通常の表として出すと意味を損なう
- 同一シート内に、役割の異なる複数の矩形領域が明確に存在する

分割方針:

- まず大きな意味のまとまりをセクション単位で切り出す
- 各セクション内で、地の文、リスト、表、画像を個別に抽出する
- セクション間は Markdown 上で見出しまたは空行で分離してよい
- セクション見出しが明確な場合は、その文言を見出しとして利用してよい
- 実装順としては、表候補抽出と地の文抽出の後、Markdown 組み立ての前に `セクション分割ブロック` を差し込むのが基本とする

出力上の考え方:

- `イベント プランナー` のようなシートでは、`議題`、`イベント チェックリスト`、`イベント カテゴリ`、`主な連絡先` のような単位へ分けることを優先する
- `老後資金プランナー` のようなシートでは、導入説明、入力ブロック、詳細表、画像を別々に扱うことを優先する
- セクション分割が有効な場合、シート全体を 1 つの巨大表として出すよりも、複数ブロックへ分解する方を優先してよい
- ただし、現時点ではフォーム風領域の扱いは保守的でよい
- 具体的には、空欄が多く罫線で構成されたフォーム風領域でも、表候補スコアが閾値を超える場合は表として残してよい
- 一方で、横に非常に広く、疎で、merge が多いフォーム領域は、巨大表化を避けるため表候補から外して地の文やセクションへ寄せてよい
- これらは将来 `フォームブロック` や `入力パネル` として別扱いにする改善余地がある

## 7. 表検出仕様

### 7.1 基本方針

表検出は全自動で行う。ただし検出結果は絶対確定ではなく、設計書として妥当な表範囲を推定するものとする。

### 7.2 主要手掛かり

表検出の主要手掛かりとして、以下を採用する。

- 値または罫線を持つ seed cell の連結成分
- 罫線
- 非空セル密度
- 先頭行のヘッダらしさ

### 7.3 罫線の役割

罫線は、表の存在検出および境界推定に使用する。

利用対象とする罫線情報:

- top
- bottom
- left
- right

考慮する観点:

- 外枠が存在する
- 内部グリッドが存在する
- 矩形としてまとまりがある
- 2 行 x 2 列以上の領域を形成する

外枠がなくても、規則性を持って線が引かれている場合は表候補として扱ってよい。

### 7.4 表検出手順

現行実装では、表検出は概ね以下の順で行う。

1. 値または罫線を持つセルを seed cell として抽出する
2. seed cell を上下左右の 4 近傍で連結成分へまとめる
3. 各連結成分の外接矩形を表候補として得る
4. 2 行 x 2 列以上などの最小条件を満たす候補を残す
5. 罫線、密度、ヘッダらしさ、結合セル過多、長文中心領域などを用いてスコアリングする

### 7.5 補助判定

必要に応じて、以下を補助情報として用いてよい。

- 非空セル密度
- 先頭行のヘッダらしさ
- 列型の一貫性

ただし、列型の一貫性や空白関係は、現行実装では強い主要判定ではなく補助的または今後強化の余地として扱う。

## 8. 表候補のスコアリング

### 8.1 基本方針

- 表検出は二値判定ではなく、表候補ごとのスコアリングで扱う
- 主要手掛かりは seed cell の連結成分としつつ、罫線・密度・ヘッダ性・列型一貫性などを補助情報として用いる
- 一定スコア以上の領域を表として採用し、それ以外は地の文候補へ回す

### 8.2 加点要素

- 外枠の罫線がある
- 内部グリッドの規則性がある
- 2 行 x 2 列以上の矩形を成す
- 非空セル密度が高い
- 先頭行がヘッダらしい短い文字列で構成される
- 列ごとの値の型や粒度に一貫性がある

### 8.3 減点要素

- 結合セルが過剰に多く、通常の表構造を崩している
- 長文セルばかりで段落ブロックに近い
- 罫線が弱く、単なる縦方向の文章列に見える

### 8.4 判定の目安

- 高スコア領域は表として採用する
- 中スコア領域は弱い表候補として扱い、将来は警告や補助表示の対象にできる
- 低スコア領域は地の文候補として扱う

### 8.5 初版の実装方針

- 初版では単純な重み付き加点方式を採用する
- 学習ベースや複雑な統計推定は行わない
- 判定理由を内部ログやデバッグ情報として保持できる設計が望ましい

## 9. 表の役割推定

### 9.1 推定結果の利用

推定した役割は、Markdown セクション見出しや内部メタ情報に反映してよい。

### 9.2 初版の扱い

初版では複雑な役割分類は必須とせず、`表001` のような連番見出しを基本とする。将来、見出し語や周辺セルから役割名を推定できるようにしてよい。

## 10. セル値処理仕様

### 10.1 通常セル

通常セルは、その値を文字列化して出力する。

### 10.2 数式セル

数式セルは可能な限り解決する。

解決対象:

- `=A1`
- `=Sheet2!B3`
- `='画面項目一覧'!C10`

非対応:

- 外部 Workbook 参照
- `='[master.xlsx]一覧'!A1`

### 10.3 数式セルの出力値決定順

数式セルの出力値は、以下の優先順で決定する。

1. 保存済み計算結果
2. 自前解決
3. 式文字列

詳細な解決経路、内部状態値、現行実装での優先順は `xlsx2md-impl-spec.md` を参照すること。

### 10.4 cached value

Excel に保存済みの計算結果が存在する場合は、それを採用する。

保存済み計算結果の具体的な判定基準や空文字キャッシュの扱いなど、実装寄りの詳細は `xlsx2md-impl-spec.md` を参照すること。

### 10.5 フォールバック

保存済み計算結果が存在しない場合は、まず自前解決を試みる。解決できない場合に限り、式文字列をそのまま出力値として採用する。空欄にはしない。

### 10.6 外部参照

外部参照は非対応とする。解決は行わず、式文字列を保持する。

### 10.7 解決状態メタ情報

数式セルについては、以下の状態を内部メタ情報として保持する。

- `resolved`
- `fallback_formula`
- `unsupported_external`

加えて、必要に応じて解決経路や保存済み計算結果の状態を区別する補助メタ情報を保持してよい。

これらの具体的な状態値、意味、現行実装での扱いは `xlsx2md-impl-spec.md` を参照すること。

## 11. 未計算 Workbook の扱い

Excel が未計算状態で保存されている場合、数式セルに対する cached value が存在しない場合がある。

この場合、本ツールは以下の動作を行う。

- まず自前解決を試みる
- それでも解決できない場合に限り、式文字列へフォールバックする
- 情報を欠落させない
- 解決できた場合は `resolved` を保持する
- 解決できない場合は `fallback_formula` を保持する

未計算 Workbook はエラーとはみなさない。

自前解決の段階構成や現行実装での evaluator / resolver の役割分担は `xlsx2md-impl-spec.md` を参照すること。

## 12. 文字列化方針

### 12.1 基本方針

Markdown 出力においては、最終的なセル値は文字列として扱う。

### 12.2 数値・日付・論理値

以下は原則として表示値相当を文字列として出力する。

- 数値
- 日付
- 真偽値

厳密な型保持は Markdown 側では要求しない。必要な内部型保持は実装側で行ってよい。

### 12.3 参照解決後の扱い

参照解決された値も、Markdown 出力時には単なる文字列値として出力する。

### 12.4 表示値優先の考え方

- `xlsx2md` の既定出力は、Excel の表示値に近い文字列を優先する
- これは既存の設計書 Excel を人間と生成 AI が共通参照する前提に合わせるためである
- 設計書レビューや問い合わせでは、Excel 上で見えている値と Markdown 上の値が近いことを重視する
- したがって、初版標準モードは `AI 向け抽象化` よりも `人間と AI の共通理解` を優先する

### 12.5 内部保持と将来モード

- 表示値を Markdown に出力する場合でも、内部的には `raw_value` を保持する
- 可能であれば `numFmtId` と `formatCode` も保持する
- 数式セルについても、cached value または再解決結果が得られた場合は、可能な範囲でセル書式を再適用し、`display` では表示値寄りに出力する
- 例: `=DATE(2024,3,17)` が日付書式セルに置かれている場合、`display` では `2024/3/17`、`raw` では `45368` として扱ってよい
- 出力モードとして、少なくとも以下を選択可能としてよい
- `display`: Excel 表示値を優先する既定モード
- `raw`: 元の内部値を優先するモード
- `both`: 表示値を本文に出しつつ、必要に応じて raw 値を補助表示するモード
- `both` の補助表示は、たとえば `2024/1/1 [raw=45292]` のような簡潔な表記としてよい
- 初版では `display` を標準としつつ、`raw` および `both` も利用可能としてよい

各モードでの細かな出力例や現行実装の UI 表示は `xlsx2md-impl-spec.md` を参照すること。

### 12.6 出力モードの具体例

```markdown
date
display: 2024/1/1
raw: 45292
both: 2024/1/1 [raw=45292]

currency
display: ¥1,024,768
raw: 1024768
both: ¥1,024,768 [raw=1024768]

fraction
display: 3/4
raw: 0.75
both: 3/4 [raw=0.75]

formula-date
display: 2024/3/17
raw: 45368
both: 2024/3/17 [raw=45368]
```

## 13. 結合セルの扱い

### 13.1 基本方針

- 結合セルは Markdown テーブル上で完全再現しない
- 結合範囲の左上セルを代表値セルとして扱う
- 結合範囲内の他セルには、空文字ではなく補助トークンを出力する
- 結合範囲そのものは内部メタ情報として保持する

### 13.2 初版の結合セル出力ルール

- 横方向結合で展開されるセルには `[MERGED←]` を出力する
- 縦方向結合で展開されるセルには `[MERGED↑]` を出力する
- 左上の代表セルには元の値を保持する
- 複合結合範囲に含まれるセルも、左上セルへの従属を示す補助トークンとして扱う

### 13.3 補足

- `[MERGED←]` および `[MERGED↑]` は、変換器が補完した構造トークンであり、元のセル値ではない
- 人間可読性を優先する将来モードでは `(同左)` や `(同上)` のような表現を選択可能としてよい

## 14. 画像対応仕様

### 14.1 対応レベル

### 14.2 対応内容

- Markdown から相対リンクする
- 画像のアンカー位置を保持する

### 14.3 保持する画像情報

### 14.4 画像位置

## 15. トレーサビリティ要件

### 15.1 基本方針

変換後 Markdown から、元の Excel 上の出典を追跡可能でなければならない。

### 15.2 理想要件

上位仕様としては、以下を少なくとも内部メタ情報または Markdown 見出し・補助セクションのいずれかで保持できることが望ましい。

- Workbook 名
- Sheet 名
- 表の Range
- 画像の Anchor
- 画像ファイルパス

### 15.3 追跡単位

- 文書単位: Workbook
- ファイル単位: Sheet
- セクション単位: 表または画像
- 領域単位: Range または Anchor

### 15.4 現行採用方式

現行実装では、Range / Anchor を体系的な共通メタフォーマットで本文全域に埋め込む方式は採らない。

代わりに、少なくとも次を明示する方式を主に採る。

- Markdown 冒頭の Workbook 名
- Markdown 冒頭の Sheet 名
- 各表見出しにおける Range 表示
- 画像・グラフ・図形セクション見出しにおける Anchor 表示

この方式により、初版では実装複雑性を抑えつつ、主要な追跡経路を確保する。

### 15.5 Workbook 名の扱い

Workbook 名はトレーサビリティ上必須のため、各 Markdown に必ず出力する。

## 16. 想定 UI

初版の UI は以下を基本とする。

1. `.xlsx` ファイル選択
2. Workbook 読み込み完了後、自動で全シートを変換
3. 変換オプション指定
4. Markdown プレビュー
5. コピーまたは保存

### 16.1 初版の主要オプション

- 先頭行をヘッダーとして扱う
- 空行を除外する
- 空列を除外する
- シート名を Markdown 見出しとして出力する
- セル内改行を `<br>` にする
- 余分な前後空白を trim する
- 出力モード: `display / raw / both`

### 16.2 UI 実装方針

- 画面実装では `lht-cmn/README.md` の方針に従う
- `md-*` を画面側で直接多用せず、原則 `lht-*` を利用する
- 単一 HTML 配布を維持しつつ、開発時は `*-src.html` と分割ソースで保守する
- Markdown を主表示とし、解析サマリーはその下へ置く
- Workbook 読み込み完了後は、自動で Markdown 生成まで進めてよい
- `解析サマリー`、`変換設定`、`表候補スコア`、`数式診断` はアコーディオンで普段は閉じてよい
- 初版では表示対象シートの切替 UI は持たず、全シート変換を基本とする
- 連結 Markdown の各チャンク先頭には、分割ファイル名ではなくチャンク識別子としての HTML コメントを付けてよい
- 例: `<!-- workbook_001_SheetName -->`

## 25. 最終要約

`xlsx2md` 初版仕様の要点は以下の通りである。

- 入力は `.xlsx` のみ対応
- 出力はシート単位の Markdown
- Workbook 名はトレーサビリティのため必須
- 表検出は、値または罫線を持つ seed cell の連結成分を起点に、罫線・密度・ヘッダ性などを併用して行う
- 地の文は表候補外の非空セル群から段落ブロックとして抽出する
- 数式は可能な限り解決し、未計算時は式文字列へフォールバックする
- 結合セルは左上代表値 + `[MERGED←]` / `[MERGED↑]` で表現する
- 外部参照は非対応
- 画像は抽出し、アンカー位置つきで保持する
- OCR や画像意味解析は初版対象外
- 出力形式は、生成 AI 入力に適した Markdown とする
