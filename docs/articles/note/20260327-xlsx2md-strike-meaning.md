## はじめに

最初は、ただ Excel を Markdown にしたかったのです。

設計書や業務資料のようなものを、あとで生成 AI に渡しやすい形にしたい。そう思うと、Excel のままより Markdown のほうが扱いやすい場面が多いです。だから `xlsx2md` というOSSアプリを作りました。

ところが、作っていく途中で、単にセルの文字列だけ取り出せばよいわけではないと気づきました。

特に気になったのが、取り消し線です。

## 最初は、ただ Excel を Markdown にしたかった

出発点はかなり素朴でした。設計書や業務資料の Excel ブックを、生成AI に渡しやすいテキストにしたかったのです。

Excel は便利です。表も書けますし、説明文も入れられますし、リンクも置けます。ただ、そのまま生成 AI に渡すには少し扱いづらいことがあります。そこで、Markdown に変換したくなりました。

この時点では、主に気にしていたのは「表が取れるか」「文章が読めるか」くらいでした。文字装飾までは、まだ本気では考えていなかった気がします。

## でも、取り消し線は落とせないと思った

少しずつ具体的な設計書のイメージを考えるうちに、取り消し線は落としてはいけない、と思うようになりました。

設計書では、取り消し線が単なる見た目ではないことがあります。

- いったん書いたが、今は採用しない
- 以前の案だが、今は無効
- 残しておくが、本文としては消したい
- 後で見直すが、このままでは使わない

こういうニュアンスが、取り消し線 1 つに乗っています。

もしここを Markdown 化の途中で普通のテキストに戻してしまうと、消されたはずの文面が、何事もなかったように生き返ります。読む人も勘違いしますし、生成 AI に渡したら、廃案の内容を有効な仕様として拾われるかもしれません。

これは、見た目が少し変わるという話ではありません。文書の意味が変わるという話です。

## 設計書では、取り消し線は状態そのもの

私はこのあたりを考えるうちに、取り消し線は装飾というより状態表示なのだ、と感じるようになりました。

太字や下線ももちろん意味を持ちます。でも取り消し線は、それより一段強く、「この記述をどう扱うか」という判断結果を背負っていることがあります。

だから、取り消し線を落とすのはまずいのです。

少なくとも、設計書を Markdown にするなら、文字列だけを抜き出して満足してはいけない。そう思うようになりました。

## とはいえ、Excel の装飾を何でも Markdown にできるわけではない

ただし、ここにはすぐ限界もあります。

Excel のほうが、Markdown よりずっと表現力が広いからです。

Excel には、太字、斜体、下線、取り消し線だけでなく、文字色、背景色、フォントサイズ、セル位置、罫線、結合、条件付き書式など、さまざまな見た目の情報があります。一方 Markdown は、もっと軽い記法です。表現できることは限られています。

なので、Excel の装飾を何でも Markdown に移せるわけではありません。完全再現はできません。

ここで必要になるのは、「どこを優先して残すか」という判断です。

`xlsx2md` では、意味を壊すとまずい装飾は優先して渡すようにしています。  
ただし、それは Markdown 側に文法や実用的な表現手段がある範囲に限られます。

この考え方にすると、無理なことを言わずに済みますし、何を守ろうとしているツールなのかも少し分かりやすくなります。

## だから `plain` と `github` を分けた

この割り切りを形にしたのが、`xlsx2md` の `plain` と `github` という 2 つの formatting mode です。

`plain` は、安全側に倒した出力です。装飾を積極的には残さず、読みやすい素直なテキストにします。Markdown の解釈差分を避けたいときや、とにかく安定してテキストにしたいときに向いています。

一方 `github` は、GitHub 互換 Markdown / HTML で表現できる範囲を使って、意味を持つ装飾をできるだけ残す出力です。

太字、斜体、取り消し線、下線、セル内改行、ハイパーリンクのように、比較的多くの対応環境で扱いやすいものは、こちらで渡します。

もちろん、これでも Excel の全部を持っていけるわけではありません。でも、少なくとも「意味を持つ装飾を平文にしてしまう」よりは、ずっとましです。

## まとめ

Excel を Markdown にしたい、と考えたとき、最初はただ文字を取り出せればよいようにも見えます。

でも実際には、設計書の中には、文字そのものだけでなく、装飾に乗った意味があります。特に取り消し線は、「この記述はそのまま使わない」という状態を表していることがあります。

だから、Markdown 化で本当に怖いのは、見た目が少し変わることではなく、意味が消えることです。

その一方で、Excel の表現力は Markdown より広いので、全部を移すことはできません。だからこそ、何を優先して残すかを考える必要があります。

`xlsx2md` は、その中で、意味を壊すとまずい装飾を優先して渡すようにしています。  
ただし、それは Markdown 側に文法や実用的な表現手段がある範囲に限られます。

地味な話ですが、この地味さこそが、設計書を次の処理へ安全に渡すうえで大事なのだと思っています。この機能追加も GPT-5.4 さんにさくっと実装していただきました。
